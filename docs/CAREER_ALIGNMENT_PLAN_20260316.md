## Biggest credibility gaps

1. **Primary vs secondary role ambiguity**
   - The package is strong, but it sometimes reads like equal parts AI engineer, architect, product builder, and experimental hacker.
   - Fix by explicitly making AI engineer the lead story and architect the support story.

2. **Reviewer overload**
   - Too many repos can dilute trust.
   - Fix by enforcing a smaller first-pass set and moving some repos into selective-use status.

3. **Data-platform signal consistency**
   - `lakehouse-contract-lab` is one of the strongest Snowflake / Databricks signals and should appear everywhere the data-platform cluster is summarized.

4. **Applied domain choice**
   - The portfolio already has strong defense, incident, semiconductor, and regulated-workflow material.
   - A new domain should only be added if it clearly strengthens the target-company narrative.

## Should a new flagship project be added?

**Not now.**

Only add a new flagship if, after tightening the front door and deepening the top repos, one gap remains clearly visible.

If a new flagship is eventually justified, the best candidate is:

- **energy / battery / grid control tower**

Why:

- it extends the existing strength in control-tower and high-trust systems,
- it reads well for Palantir + Databricks + big-tech systems roles,
- it adds a fresh domain without abandoning the current operational-software thesis.

Avoid adding a new flagship that is just another generic chat, RAG, or consumer app.
